# Backend
El backend toma el IR y genera instrucciones reales que el procesador puede ejecutar.
El backend tiene tres responsabilidades principales:

**1. Instruction Selection** → ¿Qué instrucciones de la máquina usar?
**2. Register Allocation** → ¿Dónde guardar los valores temporales?
**3. Instruction Scheduling** → ¿En qué orden ejecutar las instrucciones?


### Instruction Selection (Selección de instrucciones)
Elige las operaciones de hardware más eficientes para cada tarea de la IR. 


Para la expresión `a ← a × 2 × b × c × d`, el compilador genera algo así:

```
loadAI  rarp, @a  ⇒ ra    // cargar 'a' desde memoria al registro ra
loadI   2         ⇒ r2    // cargar la constante 2 al registro r2
loadAI  rarp, @b  ⇒ rb    // cargar 'b'
loadAI  rarp, @c  ⇒ rc    // cargar 'c'
loadAI  rarp, @d  ⇒ rd    // cargar 'd'
mult    ra, r2    ⇒ ra    // ra = a × 2
mult    ra, rb    ⇒ ra    // ra = (a×2) × b
mult    ra, rc    ⇒ ra    // ra = (a×2×b) × c
mult    ra, rd    ⇒ ra    // ra = (a×2×b×c) × d
storeAI ra        ⇒ rarp, @a   // guardar resultado en 'a'
```

Cada variable se carga desde memoria a un registro, se operan, y el resultado se guarda de vuelta.

Por ejemplo, si la maquina tiene una instruccion `add` mas rapida que `mult`, el compilador puede reemplazar `ra × 2` por `ra + ra`, que es equivalente pero más veloz. 
El selector de instrucciones conoce los trucos del procesador específico.

### Register Allocation (Asignación de registros):
Las CPUs tienen pocos "puestos de trabajo" rápidos llamados registros (una pequeña ranura de almacenamiento ultrarrápida dentro del procesador). El asignador decide qué valores se quedan en registros y cuáles se mandan a la memoria principal (proceso llamado spilling)

En el ejemplo, el allocator logra reducir de 6 a solo 2 registros reutilizándolos:

```
loadAI  rarp, @a ⇒ r1    // carga 'a' en r1
add     r1, r1   ⇒ r1    // r1 = a × 2  (usando add en vez de mult!)
loadAI  rarp, @b ⇒ r2    // carga 'b' en r2
mult    r1, r2   ⇒ r1    // r1 = (a×2) × b
loadAI  rarp, @c ⇒ r2    // reutiliza r2 para 'c'
mult    r1, r2   ⇒ r1    // r1 = (...) × c
loadAI  rarp, @d ⇒ r2    // reutiliza r2 para 'd'
mult    r1, r2   ⇒ r1    // r1 = (...) × d
storeAI r1       ⇒ rarp, @a
```

¿Cómo? 
Porque una vez que usas `b` para multiplicar, ya no lo necesitas más, entonces `r2` queda libre para cargar `c`, y luego `d`.


### Instruction Scheduling: 
Algunas operaciones, como leer de memoria, tardan varios ciclos en completarse. Como no es posible usar el resultado de una instrucción antes de que termine sin provocar esperas (ciclos perdidos), el planificador reordena las instrucciones para que la CPU ejecute otras tareas útiles mientras llegan esos datos.

El problema con el código "ingenuo":
```
Ciclo  1- 3:  loadAI @a → r1
Ciclo  4- 4:  add r1,r1 → r1        (espera a que termine loadAI)
Ciclo  5- 7:  loadAI @b → r2
Ciclo  8- 9:  mult r1,r2 → r1       (espera a que termine loadAI)
```

Total: 22 ciclos. Hay muchos ciclos desperdiciados esperando que los loads terminen.

**La solución reordenar instrucciones:** El scheduler observa que los `loadAI` son independientes entre sí. Puede arrancarlos todos al principio, en paralelo, mientras espera los resultados:

```
Ciclo 1: loadAI @a → r1    (tarda hasta ciclo 3)
Ciclo 2: loadAI @b → r2    (tarda hasta ciclo 4)
Ciclo 3: loadAI @c → r3    (tarda hasta ciclo 5)
Ciclo 4: add r1,r1 → r1    (r1 ya está listo desde ciclo 3)
Ciclo 5: mult r1,r2 → r1   (r2 listo desde ciclo 4)
Ciclo 6: loadAI @d → r2    (aprovecha que r2 ya se usó)
```

Total: 13 ciclos en vez de 22. Un 41% más rápido, sin cambiar ni una sola operación, solo reordenándolas.

<br>

**A tener en cuenta:**
Estas tres etapas se afectan mutuamente.

- El scheduling mueve loads lejos de sus multiplicaciones → los valores deben vivir en registros más tiempo → necesitas más registros.
- El register allocation asigna un valor a r2 → el scheduler no puede mover libremente las instrucciones que usan r2 → se pierde flexibilidad.

No hay una solución perfecta y universal. El compilador tiene que hacer compromisos de ingeniería entre velocidad, uso de registros y complejidad del análisis. Por eso dice el texto que los problemas más difíciles de compiladores ocurren en el back end.
