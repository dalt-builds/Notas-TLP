# El optimizador
El optimizador toma el IR que el frontend genero y en el busca redudancias, frases innecesarias, busca caminos mas cortos y efecitvos, con el objetivo de hacer mas eficiente y rapido el codigo, pero manteniendo su signficado

## Analisis:
El compilador examina el código IR y recolecta hechos. Se mencionan dos tipos de análisis:

- **Data-flow analysis:** Razona sobre qué valores fluyen a través del programa en tiempo de ejecución, pero lo hace en tiempo de compilación. 
Por ejemplo, detecta que `b` y `c` no reciben ningún nuevo valor dentro del loop.

- **Dependence analysis:** Analiza expresiones matemáticas para determinar qué partes del código dependen de qué otras, especialmente útil con arreglos.

## Transformación:
Una vez que el análisis confirma que es seguro y rentable mover `2 × b × c` fuera del loop, el compilador reescribe el IR para hacerlo.
La transformación específica que ocurre aquí se llama **loop-invariant code motion** (movimiento de código invariante fuera del ciclo). Es una de las optimizaciones más comunes y poderosas.

**Ejemplo:**
Imagina este código dentro de un bucle que se repite *n* veces:

- **Original (IR inicial):**
```
for i = 1 to n
    a ← a × 2 × b × c × d
```
Aquí se hacen `4 × n` multiplicaciones

- **Optimizado:** El optimizador analiza que `2 × b × c` nunca cambia dentro del bucle
Entonces lo transforma así:
```
t ← 2 × b × c  // Cálculo movido fuera
for i = 1 to n
    a ← a × d × t
```
Ahora solo se hacen `2 × n + 2` multiplicaciones, lo cual es mucho más rápido si n es grande

### Resumen visual

```
IR sin optimizar
      ↓
  [Análisis]      → ¿qué sé sobre los valores en cada punto del código?
      ↓
 [Transformación] → reescribir el IR de forma más eficiente
      ↓
  IR optimizado   → mismo resultado, menos trabajo
```
